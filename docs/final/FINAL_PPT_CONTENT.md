## 0. PPT 总体定位

### 项目一句话

PocketOps 是面向工业车辆维保现场的端侧 AI 诊断助手，在无公网环境下，通过手机端 NPU、本地知识图谱和多模态识别完成故障诊断、现场追问、维修建议、动态工单与离线提交闭环。

### 30 秒版本

工业车辆维保现场经常面临网络不可用、资料分散、经验依赖强和工单记录不完整的问题。PocketOps 把本地 GraphRAG 知识图谱、端侧大模型、多模态图片/视频识别和工单闭环放进手机。工程师可以输入故障、上传零件/铭牌/现场照片或视频，系统在端侧给出诊断依据、备件、步骤、相似工单和待补充信息，并生成可提交的工单。电脑服务只作为局域网资料与工单归档服务，诊断主链路不依赖公网。

### 3 个核心卖点

1. 端侧可用：模型服务运行在手机本机 `127.0.0.1:8910`，弱网或无公网仍可完成诊断。
2. 图谱定事实：设备、症状、原因、步骤、备件和相似工单优先来自本地知识图谱，减少大模型幻觉。
3. 工单闭环：从诊断、追问、补参数、排查清单全选到动态工单、质量检查、离线队列和局域网补交，形成现场业务闭环。

### 建议答辩主线

背景痛点 -> PocketOps 方案 -> 技术架构 -> 骁龙端侧部署 -> 现场演示路径 -> 完成度与验证 -> 落地价值。

### 建议视觉风格

- 风格：工业现场、移动端工作台、技术可信、少营销化。
- 主色：深墨绿 / 石墨黑 / 白底科技蓝均可，避免过度炫彩。
- 图形：用流程图、设备图、手机界面截图、知识图谱节点图、工单表单图。
- 每页密度：标题 1 句，主文案 3-5 条，右侧放图或流程。

## 4. 项目背景与应用场景

对应模板：Slide 4。

### 页面标题

工业车辆维保现场：网络弱、资料散、经验强依赖

### 应用场景与目标用户

- 场景：仓储、制造、物流园区中的叉车、AGV、搬运车等工业车辆维保。
- 用户：一线维修工程师、售后服务人员、设备主管、维保外包团队。
- 现场特点：设备分散、网络不稳定、故障信息多源、维修过程需要留痕。

### 典型痛点

- 弱网环境：现场常在仓库、厂区、地下或屏蔽区域，公网不可用或不稳定。
- 资料分散：维修手册、历史工单、备件信息和设备铭牌分散在不同系统或纸质资料中。
- 经验依赖：新手维修人员难以快速判断排查顺序，老工程师经验难复用。
- 工单质量不稳定：设备编号、故障现象、参数、处理步骤经常缺失，后续复盘困难。
- 图片视频未结构化：现场拍摄的故障点、铭牌、仪表和零件照片难以直接进入诊断流程。

### 项目切入点

把“现场诊断 + 多模态采集 + 知识图谱 + 工单闭环”放进手机端，让工程师在离线或局域网条件下也能完成关键工作。

### 价值主张

- 对一线人员：减少查资料和问专家时间，提供可执行排查步骤。
- 对服务组织：提升工单完整度，沉淀结构化维修知识。
- 对设备企业：降低售后响应成本，提升现场诊断一致性。

### 图示建议

左侧：工业车辆现场照片 / 叉车图标。  
右侧：痛点矩阵，四个卡片：弱网、资料散、经验依赖、工单缺项。  
底部：PocketOps 切入点箭头：现场输入 -> 端侧诊断 -> 动态工单。

### 讲稿

工业车辆维修不是单纯问一个模型就结束。现场要先确认设备、看铭牌和零件、读取参数、查历史案例，再形成工单。我们选择从一线维保人员的手机入手，把这些动作串成一个离线可用的工作流。

## 5. 方案简介与核心价值

对应模板：Slide 5。

### 页面标题

PocketOps：端侧 AI + 本地知识图谱 + 动态工单闭环

### 方案概述

PocketOps 是一个 Android 端工业车辆诊断应用。用户可以通过文字、图片、视频或模拟诊断参数描述故障，应用优先在本地知识图谱中匹配设备、症状、原因、备件、步骤和相似工单；需要补充判断时调用手机本机 NPU 推理服务。最终输出诊断卡片、排查清单和动态工单，并支持离线队列和局域网补交。

### 核心亮点 / 创新点

1. 离线端侧诊断：本机知识图谱 + 本机 HTTP 推理服务，不依赖公网完成主诊断流程。
2. 图谱增强生成：图谱锁定事实，大模型只补充推理依据、风险和待采集信息。
3. 多模态现场输入：支持文字、零件图片、铭牌图片、现场视频点检和模拟故障码/参数。
4. 动态工单闭环：连续问答、图片分析、视频点检结论、设备上下文、用户补充参数和已完成排查项被汇总进工单。
5. 局域网资料与工单服务：电脑侧 demo server 可提供同步、资料下载、工单归档。

### 功能模块 1：端侧诊断工作台

- 解决问题：弱网环境下无法调用云端诊断。
- 展示效果：登录后加载知识库和本机推理服务，顶栏显示就绪状态。

### 功能模块 2：多模态与图文追问

- 解决问题：铭牌、零件、仪表、故障点等现场信息无法结构化使用。
- 展示效果：上传零件图片问“这是什么零件”；上传铭牌后追问配置号和叉车编号；上传环车视频后输出点检结论、仪表盘异常判断、环车外观风险、处理建议和需补拍内容。

### 功能模块 3：知识图谱与工单闭环

- 解决问题：诊断缺依据、维修步骤不统一、工单信息不完整。
- 展示效果：诊断卡片、图谱依据、相似工单、离线排查清单；问诊/泛化清单全选后出现“生成动态工单”，并进入质量检查。

### 页面文案精简版

- 输入：文字故障、图片/视频、铭牌、模拟故障码和关键参数。
- 处理：本地 GraphRAG 优先，端侧模型补充推理，多轮上下文和排查清单持续汇总。
- 输出：诊断卡片、维修步骤、备件、相似工单、排查清单、动态工单、离线提交队列。

### 讲稿

PocketOps 的核心不是单个聊天机器人，而是一个现场工作台。它既能看图片、文字和环车点检视频，也能把结果落到维修步骤和工单里。视频点检会输出完整点检报告，覆盖仪表盘识别汇总、故障码汇总、反光镜检查、环车外观风险、处理建议和需补拍内容。我们用知识图谱保证事实来源，用端侧模型补充推理解释，并在排查清单完成后把用户回答和已完成检查项写进动态工单。

### 技术栈

- Android：Kotlin + Jetpack Compose + Material 3。
- 端侧服务：Android Foreground Service `GenieHttpService`。
- 本机接口：`GET /v1/models`、`POST /v1/chat/completions`、`POST /clear`。
- 推理模型：`qwen2.5vl-3b-8850-2.42`。
- 知识图谱：APK 内置 `assets/maintenance/knowledge_graph.json`，当前样例包含 84 个节点、84 条边。
- 图谱类型：设备、故障症状、根因、备件、维修步骤、人员、历史工单。
- 工单服务：局域网 demo server，支持登录、manifest、资料查询、工单提交与查询。

### 关键工程方法

- 启动时先同步 / 加载知识图谱，再检查本机推理服务。
- 如果本机服务未就绪，应用拉起前台服务并轮询 `/v1/models`。
- 每次诊断前调用 `/clear` 清理服务侧对话状态。
- 文本推理按流式响应处理；图片和视频走非流式多模态请求。
- 视频先抽关键帧并拼接为 contact sheet，再统一送入 VLM；车辆点检视频输出完整点检报告，包含仪表盘识别汇总、故障码汇总、仪表盘异常判断、反光镜检查、环车外观风险、处理建议和需补拍内容。
- 动态工单不是复制单条回答，而是汇总当前连续上下文。

### 端云协同口径

- 主诊断链路：端侧。
- 可选服务：局域网电脑服务，用于资料同步、附件下载、工单归档。
- 不依赖公网完成：文字诊断、图片/视频分析、图谱查询、工单草稿、历史回看。

### 讲稿

这页展示整体架构。PocketOps 的核心链路在手机端：本地知识图谱负责确定性事实，NPU 模型负责多模态和推理补充，客户端负责结果结构化和工单闭环。电脑侧不是云端诊断，只是局域网资料和工单服务，可用时增强归档，不可用时不影响主诊断。

## 7. 高通技术应用与设备部署

对应模板：Slide 7。

### 页面标题

骁龙端侧部署：把多模态诊断放在移动设备上运行

### 已使用的高通相关能力

- 平台能力：面向骁龙移动平台的端侧 AI 推理能力。
- 推理位置：本机 NPU / 端侧推理服务。
- NPU 证明方式：导出 `adb logcat`，用 HTP / CDSP 关键日志证明推理走 Qualcomm QNN HTP。
- 运行方式：Android 应用内检查并拉起本机 Genie HTTP 推理服务。
- 本机端口：`127.0.0.1:8910`。
- 模型目录：`/sdcard/GenieModels/<model-dir>/config.json`。
- ABI：`arm64-v8a`。

### 模型与部署信息

- 当前运行模型名：`qwen2.5vl-3b-8850-2.42`。
- 支持能力：文本诊断、图片识别、视频关键帧分析、视觉上下文追问。
- 请求方式：OpenAI-compatible `/v1/chat/completions`。
- 状态检查：`/v1/models` 返回模型元数据后认为 ready。
- 前台服务：`GenieHttpService` 常驻，异常退出后可重试拉起。

### Android 运行环境

- 应用名称：PocketOps。
- applicationId：`com.pocketops.app`。
- versionName / versionCode：`4.8 / 48`。
- minSdk / targetSdk / compileSdk：`31 / 35 / 35`。
- Android 入口：`PocketOpsActivity`。
- 推理服务：`GenieHttpService`。
- 设备要求：Android 12+、arm64-v8a、可读取模型目录。

### 手机 NPU / HTP 日志验证页内容

这一部分建议在正式 PPT 中单独做一页，标题可用：`手机端 NPU HTP 使用验证`。

#### 验证流程

```bash
adb devices
adb logcat -c
adb logcat > log.txt
# 在手机上打开 PocketOps，触发文字诊断、图片诊断或视频分析
# 推理完成后按 Ctrl+C 停止日志
```

Windows PowerShell 快速验证：

```powershell
Select-String -Path log.txt -Pattern "libQnnHtp|remote_handle64_open|Successfully opened.*Htp|_dom=cdsp|domain 3"
```

#### PPT 页面要放的关键证据

| 日志关键字 | 证明含义 |
| --- | --- |
| `libQnnHtp` / `libQnnHtpV81Skel.so` | QNN HTP 相关库被加载 |
| `Successfully opened file ... libQnnHtpV81Skel.so` | HTP skel 库文件成功打开 |
| `remote_handle64_open` | DSP 远程句柄建立成功 |

- 成功：日志同时包含 `remote_handle64_open`、`_dom=cdsp`、`domain 3`。

- 失败：没有 HTP 相关日志，或出现 `Permission denied` / `Error 0xe`，需要检查设备权限、模型库和 APK 是否触发推理。

### 为什么适合骁龙平台

- 实时性：现场交互需要低延迟反馈，端侧推理减少网络往返。
- 弱网可用：仓储和厂区网络不稳定，端侧模型保障主流程可运行。
- 隐私与安全：设备编号、故障码、维修记录等敏感数据可留在本机或局域网。
- 多模态移动性：工程师用手机直接拍照、录视频、生成工单，符合现场工作方式。
- 计算效率：NPU 加速适合移动端多模态模型的持续交互。

## 15. 答辩问答准备

### Q1：为什么一定要端侧，而不是云端大模型？

工业车辆维保现场经常处于弱网或无公网环境。端侧部署可以保障诊断主流程可用，减少网络延迟，同时设备编号、故障码、维修记录等敏感数据可以留在本机或局域网内。

### Q2：如何降低大模型幻觉？

PocketOps 不是让大模型直接编结论。命中知识图谱时，设备、症状、根因、备件、步骤和相似工单由图谱确定；端侧模型只补充推理依据、风险提示和待采集信息。未命中时也会明确提示“知识库未直接命中”，只给泛化排查路径。

### Q3：知识图谱现在规模多大？后续如何扩展？

当前演示样例包含 84 个节点和 84 条边，覆盖设备、症状、根因、备件、维修步骤、人员和工单。后续可以从企业设备台账、维修手册、历史工单和备件库中持续扩展，并通过远程 manifest 同步到客户端。

### Q4：蓝牙诊断是真实接入了吗？

当前决赛演示中的蓝牙诊断采集是模拟故障码和关键参数，用于展示故障码/参数如何进入端侧诊断链路。真实车载控制器接入属于后续迭代，需要按具体车辆协议和外设进行适配。

### Q5：电脑侧服务不可用怎么办？

诊断主链路仍在手机端可用。工单提交失败时会进入本机待提交队列，服务恢复后可以补交；演示或测试工单也可以手动清空。

### Q6：如何体现骁龙平台优势？

PocketOps 的核心场景是移动端现场诊断。骁龙平台提供端侧 AI 推理、多模态处理和移动设备能力，使工程师能直接用手机拍照、录视频、读取现场信息并生成结果，在弱网环境下仍保持交互连续性。PPT 中会补充手机 NPU HTP 日志截图，用 `remote_handle64_open`、`_dom=cdsp`、`domain 3` 证明推理实际建立了 CDSP/HTP 通道。

### Q6-1：如何证明不是 CPU 在跑，而是 NPU HTP 被调用？

我们按 `NPU_HTP_日志导出指南.md` 使用 `adb logcat` 导出手机日志。推理前先执行 `adb logcat -c` 清空缓冲区，再运行文字或图片诊断，最后搜索 `libQnnHtp`、`remote_handle64_open`、`_dom=cdsp` 和 `domain 3`。其中 `remote_handle64_open` + `_dom=cdsp` + `domain 3` 是关键证据，表示 QNN HTP skel 库和 Hexagon DSP/CDSP 通道建立成功。

### Q7：项目完成度如何？

当前已完成 Android 主线、本机推理服务接入、本地知识图谱、多模态诊断、端侧问诊、排查清单生成动态工单、离线队列、历史搜索和局域网 demo server。决赛前主要工作是补充性能截图、导出 HTP 日志、验证比赛设备、准备演示素材和备用录屏。

### Q8：和普通 ChatGPT / 拍照识别有什么不同？

PocketOps 面向工业车辆维保业务流，不只是回答图片内容。它能结合本地知识图谱、历史工单、维修步骤、备件、设备编号和工单质量检查，把诊断结果转成可执行、可留痕、可提交的维修工单。

### Q9：数据安全如何考虑？

主诊断链路运行在手机本机，资料同步和工单归档可部署在局域网。敏感数据不需要上传公网云服务。服务端接口也设计了 token、tenant/site、文件哈希和下载鉴权等机制。

### Q10：如果知识库没有命中，系统还能用吗？

可以。系统会明确进入 `知识库未命中 · 端侧泛化`，给出假设、排查路径和需要补采的证据。最新版流程要求先完成下方排查清单，清单全选后才出现“生成动态工单”，这样工单会包含用户描述和已完成检查项，同时提示缺少设备编号、故障码或触发工况，避免误导为确定结论。
